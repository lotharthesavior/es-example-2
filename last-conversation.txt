❯ there should never be a model call from controllers - for proper layer        
  separation. Therefore, adjust the controllers so we have that delegated to    
  repository classes instead. The controller would pass a DTO to the service,   
  extracted directly from a dedicated method from the custom request objects,   
  right into the repository, the commandbus will be called from there, giving   
  back the content to be passed to the response. You created a "Response" to    
  give back responses, which is dumb, Laravel has Resource abstraction to take  
  care of that, change to that instead erasing this custom object abstraction.

     Refactor: Layered Architecture (DTO → Service → Repository → Resource)

     Context

     DemoController.php and DashboardController.php reach into Eloquent
     (Product::, Cart::, Order::, EloquentStoredEvent::) and dispatch CommandBus
      directly, mixing HTTP, persistence, and command-dispatch concerns. A
     custom App\Http\Responses\ApiResponse hierarchy (12 classes) duplicates
     what Laravel's JsonResource already provides.

     Target: controllers contain only request → service → resource. Repositories
      own all model access and command dispatch. Fallback-resolution lives in
     repositories. Custom Response abstraction is deleted in favour of Laravel
     API Resources.

     Target data flow

     HTTP → FormRequest::toDto() → Controller → Service → Repository (CommandBus
      + queries) → Model|Result DTO → JsonResource → JsonResponse

     After refactor: zero Model:: or EloquentStoredEvent:: calls in either
     controller; zero commandBus->dispatch(...) outside repositories.

     public function toDto(): AddToCartData

             cartUuid:  $v['cart_uuid'] ?? null,
             quantity:  (int) ($v['quantity'] ?? 1),
         );
     }

     Payload contract — preserved vs lost

     Preserved (FE depends on these keys, verified via grep over
     resources/views/dashboard.blade.php):

     - Action POST responses: cart_uuid, order_uuid
     - /api/state cart: uuid, status, totalFormatted, itemCount, 
     items[].{productName,quantity,lineTotal}
     - /api/state order: uuid, status, totalFormatted, cancellationReason, 
     paymentReference, trackingNumber
     - /api/stats: totalEvents, activeCarts, ordersToday, revenueToday
     - EventHistoryResponse shape: order_uuid, at, event_count, history[]
     - Error: {"error": "..."} with 422

     Lost (none read by FE):

     - message field on every action response (was cosmetic)
     - product_name on add-to-cart response
     - snake-case payment_reference, tracking_number, reason, amount at top
     level of action responses — FE re-fetches from /api/state after every
     action, so the camelCase keys inside OrderResource cover the read path.

     Risks

     - PHPStan level 6 + larastan. Resources receive mixed $this->resource; add
     @property-read Cart $resource etc. on each entity resource, or override the
      constructor signature to type the input.
     - Order projection timing in CheckoutResult.order. Relies on synchronous
     projector execution. Verify config/event-sourcing.php has no async queue;
     if async, the resource must tolerate null.
     - tests/Feature/OrderFlowTest.php:36 — only test asserting on a deleted key
      (product_name/message). Update as noted above.
     - Repository layering inversion. Putting CommandBus inside repositories is
     unusual (write-side concern in a typically-read-side abstraction).
     Acceptable per explicit request; revisit if write-side grows beyond
     CRUD-style action endpoints.
